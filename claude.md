# Claude Transcript：AI Coding Agent 项目（claude-code-mini）

> 项目名称：`claude-code-mini`  
> 项目定位：一个可运行、可扩展、可复现的 CLI Agent 工程项目  
> 协作方式：开发者主导 + AI 编程助手辅助实现与审查  
> 记录范围：从最小可用版本到完整工程版本的落地过程

---

## 任务目标

1. 从零实现最小 Agent 主循环，能稳定完成“模型决策 -> 工具执行 -> 结果回灌”。
2. 按真实开发痛点逐步扩展能力：规划、上下文管理、任务持久化、异步执行、多代理协作等。
3. 形成可复现工程：
   - 可本地启动；
   - 能按阶段验证；
   - 有统一入口可直接体验完整能力。

---


### Step 1：最小可运行版本

- 修改文件：`agents/s01_agent_loop.py`
- 修改细节：
  - 建立核心 while 循环：请求模型、识别 `tool_use`、执行工具、回写 `tool_result`。
  - 先只接一个 `bash` 工具，验证最小闭环。
  - 补充 CLI 交互入口，支持连续输入。
- 验证方式：
  - 启动脚本并发出简单命令，确认模型可触发工具且返回结果。

### Step 2：工具层工程化

- 修改文件：`agents/s02_tool_use.py`
- 修改细节：
  - 引入 `TOOL_HANDLERS`（工具名到处理函数映射）。
  - 引入 `TOOLS` schema（工具声明与参数约束）。
  - 增加 `read_file/write_file/edit_file`，形成文件级开发能力。
- 验证方式：
  - 连续执行读写编辑操作，确认工具路由与参数解析正确。

### Step 3：引入执行计划（Todo）

- 修改文件：`agents/s03_todo_write.py`
- 修改细节：
  - 增加 Todo 工具与状态流转（pending/in_progress/completed）。
  - 强制先列计划再执行，降低任务漂移。
  - 在对话中可持续追踪执行进度。
- 验证方式：
  - 多步任务下检查是否能先拆分再逐步完成，不遗漏核心步骤。

### Step 4：引入子代理分治

- 修改文件：`agents/s04_subagent.py`
- 修改细节：
  - 支持主代理创建子代理处理子任务。
  - 子代理使用独立上下文，主上下文仅接收结果摘要。
  - 降低复杂任务对主上下文的污染。
- 验证方式：
  - 给出包含多个子问题的大任务，确认子任务可独立完成并回传。

### Step 5：按需知识加载

- 修改文件：`agents/s05_skill_loading.py`
- 修改细节：
  - 支持运行时加载 `SKILL.md`，而非一次性塞满 system prompt。
  - 技能内容通过工具结果注入当前任务上下文。
  - 降低 token 消耗并提升知识命中率。
- 验证方式：
  - 在无技能与有技能两种场景对比输出质量与上下文长度。

### Step 6：上下文压缩与续航

- 修改文件：`agents/s06_context_compact.py`
- 修改细节：
  - `micro_compact`：压缩旧工具结果，保留近期关键信息。
  - `auto_compact`：超过阈值后自动生成摘要，并将完整会话写入 `.transcripts`。
  - `compact` 工具：支持手动压缩。
- 验证方式：
  - 构造长会话触发自动压缩，确认压缩后仍可延续上下文任务。

### Step 7：任务持久化系统

- 修改文件：`agents/s07_task_system.py`
- 修改细节：
  - 引入 `TaskManager`，任务落盘到 `.tasks/task_*.json`。
  - 支持 CRUD 与 `blockedBy` 依赖关系。
  - 上游完成后自动清理下游阻塞依赖。
- 验证方式：
  - 创建依赖链任务，逐步完成并检查依赖自动解除是否正确。

### Step 8：后台异步执行

- 修改文件：`agents/s08_background_tasks.py`
- 修改细节：
  - 新增 `BackgroundManager`，后台线程执行耗时命令。
  - 新增 `background_run/check_background` 工具。
  - 每轮模型调用前注入后台任务完成通知（`drain_notifications`）。
- 验证方式：
  - 启动长耗时命令并继续进行其他对话，检查是否不阻塞主流程。

### Step 9：多代理协作

- 修改文件：`agents/s09_agent_teams.py`
- 修改细节：
  - 构建主代理 + 成员代理的协作框架。
  - 增加代理间消息交换与任务分工机制。
  - 支持并行处理子任务。
- 验证方式：
  - 把一个复杂需求拆成并行子任务，确认汇总输出完整性。

### Step 10：团队协议规范化

- 修改文件：`agents/s10_team_protocols.py`
- 修改细节：
  - 统一请求、响应、确认、收敛的通信规范。
  - 让多代理协作从“能跑”升级到“可控”。
  - 减少状态错乱与重复沟通。
- 验证方式：
  - 在多轮协作中观察状态是否一致、交付是否按协议收敛。

### Step 11：自治化执行

- 修改文件：`agents/s11_autonomous_agents.py`
- 修改细节：
  - 增加空闲轮询与自动认领任务能力。
  - 从“主代理手工分配”升级到“代理自主推进”。
  - 提升系统长期运行效率。
- 验证方式：
  - 不人工逐条派发任务，检查代理是否可自主推进任务板。

### Step 12：任务与目录隔离

- 修改文件：`agents/s12_worktree_task_isolation.py`
- 修改细节：
  - 任务执行与隔离目录绑定，避免并行覆盖冲突。
  - 提升并行开发稳定性与可回滚能力。
  - 为复杂团队协作提供安全执行边界。
- 验证方式：
  - 多任务并发执行，检查相互文件影响是否被隔离。

### Step 13：完整版本整合

- 修改文件：`agents/s_full.py`
- 修改细节：
  - 汇总前述能力为统一入口。
  - 对工具、任务、上下文、协作能力进行统一编排。
  - 作为可直接体验的生产化原型脚本。
- 验证方式：
  - 通过一个多步骤真实任务进行端到端演练。

### Step 14：复现文档与运行说明

## 3. 修改原因

### 架构层原因

- 我采用渐进式架构：先保证核心环可运行，再逐步添加复杂能力，避免一开始过度设计。
- 每步只解决一个主要问题（如“上下文爆炸”“阻塞执行”“多代理混乱”），降低改动风险。

### 工程层原因

- 用文件化任务和 transcript 持久化，便于调试、回放与问题复盘。
- 用工具注册和 schema 约束，控制扩展复杂度并提升可维护性。
- 用隔离执行与协议治理，提升并行场景下的稳定性。

### AI 协作层原因

- 我把 AI 当“实现加速器 + 初审器”，而不是最终决策者。
- 关键逻辑（状态机、并发、边界条件）坚持人工把关。
- 通过“先验收标准、再实现、再回归”的节奏控制质量。

## 4. 最终结果

### 4.1 交付结果

- 交付了一个自建、可本地复现、可持续迭代的 AI Agent CLI 工程项目。
- 形成从 `s01` 到 `s12` 的分阶段实现，以及统一整合版本 `agents/s_full.py`。
- 提供完整文档链路，支持他人快速理解并复现实验。

### 4.2 技术结果

- 已具备核心能力：工具系统、计划系统、子代理、技能加载、上下文压缩、任务持久化、后台任务、多代理协作、协议治理、自治执行、目录隔离。
- 系统具备工程化特征：模块清晰、可扩展、可观测、可回放。

### 4.3 可复现方式（简要）

1. 安装依赖并配置 `.env`。  
2. 逐步运行 `agents/s01_agent_loop.py` 到 `agents/s12_worktree_task_isolation.py`。  
3. 最后运行 `agents/s_full.py` 做端到端验证。  
4. 查看 `.tasks` 与 `.transcripts`，验证持久化与压缩链路。
